Bloco 01 — Processos

Programa vs Processo: a Diferença Vital

  • Programa: arquivo estático em disco (código + dados)
  • Processo: programa em execução com estado próprio
  • Dois cliques no mesmo .exe → dois processos distintos
  • Cada processo tem seu próprio espaço de endereçamento
  • O SO gerencia dezenas de processos simultaneamente

Bloco 01 — Processos

Por Dentro de um Processo: o PCB

  • PCB = Process Control Block: "ficha" do processo no SO
  • Contém: estado atual, program counter, registradores
  • Também: informações de memória, I/O, escalonamento
  • O SO usa o PCB para pausar e retomar um processo
  • Troca de contexto = salvar PCB A, carregar PCB B

Bloco 01 — Processos

Modelo de 5 Estados do Processo

  • new → processo criado
  • ready → pronto para executar
  • running → executando no CPU
  • waiting → aguardando evento (I/O)
  • terminated → finalizado

Bloco 01 — Processos

Troca de Contexto (Context Switch)

  • Interrupção do SO: timer ou evento força parada
  • Salvar estado: registradores e PC no PCB do processo atual
  • Restaurar novo: carregar registradores e PC do próximo processo
  • Overhead: centenas de microsegundos — impacto real em desempenho

Bloco 02 — Threads

Thread vs Processo — Espaço de Endereçamento

  • Processo: heap, dados e stack isolados (espaço próprio)
  • Thread: múltiplas dentro de 1 processo, compartilham heap e dados
  • Implicação: threads acessam a mesma variável simultaneamente
  • Consequência: sincronização necessária; isolamento de processos é seguro

Bloco 02 — Threads

O Que Threads Compartilham vs Privado

Compartilhado (todas as threads)
  • Código do programa
  • Heap (variáveis dinâmicas)
  • Dados globais
  • Descritores de arquivo
Privado / por thread
  • Stack (variáveis locais)
  • Program Counter (PC)
  • Registradores da CPU
  • Flags de status
Global → sincronizar
counter = 0 # global → sincronizar!
Local → thread-safe
x = 0 # local → thread-safe

O Problema — Contador Compartilhado

  • 9 threads incrementam a mesma variável 1.000.000 vezes cada
  • Esperado: counter = 9 × 1.000.000 = 9.000.000
  • Real: counter ≈ 6.234.127 (diferente a cada execução)
  • Por quê? Leitura simultânea corrompe o valor

Bloco 03 — Condição de Corrida

Por Que counter += 1 Não é Atômico?

  • counter += 1 = READ + ADD + WRITE (3 operações separadas)
  • Thread A e Thread B podem intercalar no momento errado
  • Interleaving: A lê (5), B lê (5), A escreve (6), B escreve (6)
  • Resultado: ambas escrevem 6 — um incremento perdido!

Bloco 03 — Condição de Corrida

Seção Crítica e Exclusão Mútua

  • Seção crítica: trecho de código que acessa recurso compartilhado
  • Problema: duas threads na seção crítica = race condition
  • Exclusão mútua: apenas uma thread por vez na seção crítica
  • Requisito: lock/mutex garante acesso exclusivo

Bloco 03 — Condição de Corrida

Lock/Mutex em Python — threading.Lock

  • lock = threading.Lock() cria objeto lock
  • lock.acquire() — tenta adquirir, bloqueia se ocupado
  • lock.release() — libera para a próxima thread
  • with lock: — alternativa pythônica (libera automaticamente)
SEM Lock
counter = 0 def incrementar(): global counter for i in range(100000): counter += 1 # race!
≈ 6,234,127 ✗
COM Lock
counter = 0 lock = threading.Lock() def incrementar(): global counter for i in range(100000): lock.acquire() counter += 1 lock.release()
900,000 ✓
Pythônico: with lock: counter += 1 ← libera automaticamente

Bloco 04 — GIL e Distribuído

Python GIL — Global Interpreter Lock

  • GIL: mutex global que permite apenas 1 thread Python executar por vez
  • Razão: gerenciamento de memória interna simplificado
  • CPU-bound: threads NÃO paralelizam (GIL força execução serial)
  • IO-bound: threads SIM paralelizam (GIL liberado durante I/O)

Bloco 04 — GIL e Distribuído

De Memória Compartilhada a Distribuída

  • Threads: compartilham memória, precisam de locks
  • Processos isolados: sem compartilhamento, sem race conditions
  • Problema: processos isolados precisam se comunicar
  • Solução: sockets (TCP/IP) como canal entre processos

Fluxo TCP — bind/listen/accept/send/recv

  • Cliente: socket() → connect() → conexão estabelecida
