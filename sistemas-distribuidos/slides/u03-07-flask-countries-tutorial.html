O que vamos construir

Uma API REST que gerencia uma lista de países. Você vai construir três versões do mesmo serviço, cada uma mais robusta que a anterior — da mais simples (tudo em memória) até a que usa ORM e serialização automática.

01

API em Memória

Lista Python. Simples, sem persistência. Entenda rotas e JSON.

02

Persistência SQLite

Banco de dados real. db.py abstrai cursor e conexão.

03

SQLAlchemy + Marshmallow

ORM + schemas. CRUD completo com serialização automática.

Pré-requisitos

  • Python 3.10 ou superior
  • Ter assistido ao slide 6 — REST com Flask/Python
  • Saber fazer requisições HTTP (métodos, cabeçalhos, body JSON)

Etapa 1 — API em memória

O ponto de partida mais simples possível. Os dados ficam em uma lista Python — sem banco, sem arquivo. Perfeito para entender rotas e serialização JSON.

Figura 2 — Ciclo de requisição na Etapa 1. Dados residem na memória do processo.

GET /countries

@app.route("/countries", methods=['GET'])
def get_countries():
    return jsonify(countries)

Retorna a lista completa em formato JSON com status 200.

POST /countries

@app.route("/countries", methods=['POST'])
def add_country():
    if request.is_json:
        country = request.get_json()
        country["id"] = _find_next_id()
        countries.append(country)
        return country, 201
    return {"error": "..."}, 415
Limitação importante: ao reiniciar o servidor, todos os dados somem. A memória do processo não é persistente. — Esta é a motivação para a Etapa 2.

Testar com curl

# Adicionar país

Etapa 2 — Persistência com SQLite

Agora os dados sobrevivem ao restart. Usamos o módulo sqlite3 da biblioteca padrão, encapsulado em db.py para separar responsabilidades.

Figura 3 — Etapa 2: Flask delega acesso a dados para db.py, que gerencia a conexão SQLite.

schema.sql — estrutura da tabela

DROP TABLE IF EXISTS countries;

CREATE TABLE countries (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  country_name TEXT    UNIQUE NOT NULL,
  capital      TEXT    UNIQUE NOT NULL
);

Inicializar o banco

# Dentro da pasta 02-sqlite/:
flask --app app init-db

O comando init-db é registrado via app.cli.add_command em db.py.

Padrão de conexão por request

def get_db():
    if 'db' not in g:
        g.db = sqlite3.connect(DATABASE, detect_types=...)
        g.db.row_factory = sqlite3.Row  # retorna dict-like
    return g.db
g é o objeto de contexto do Flask: vive durante um único request e é limpo automaticamente no teardown_appcontext.

Etapa 3 — SQLAlchemy + Marshmallow

ORM completo com serialização automática. Você define modelos como classes Python e o SQLAlchemy cuida do SQL. Marshmallow transforma objetos em JSON e valida dados de entrada.

Figura 4 — Etapa 3: ORM abstrai SQL; Marshmallow serializa/deserializa JSON automaticamente.

Modelo (ORM)

class Country(db.Model):
    id           = db.Column(db.Integer,
                     primary_key=True)
    country_name = db.Column(db.String(100))
    capital      = db.Column(db.String(100))

Schema (Marshmallow)

class CountrySchema(ma.SQLAlchemyAutoSchema):
    class Meta:
        model = Country
        load_instance = True

country_schema  = CountrySchema()
countries_schema = CountrySchema(many=True)

Inicializar e rodar

python 03-sqlalchemy/app.py
O banco é criado automaticamente via db.create_all() dentro do contexto da aplicação — não precisa de init-db manual.

CRUD completo disponível nesta etapa

Além de GET e POST, agora temos PUT e DELETE por ID:

curl -X PUT http://localhost:8090/countries/1 \
  -H "Content-Type: application/json" \
  -d '{"country_name":"Thailand","capital":"Bangkok"}'

curl -X DELETE http://localhost:8090/countries/1

Cliente Python

client.py demonstra como consumir a API usando a biblioteca requests — a mesma forma que qualquer sistema distribuído cliente usaria. 

import requests

api_url = 'http://127.0.0.1:8090/countries'

def get():
    response = requests.get(api_url)
    print(response.json())

def insert():
    country = {"country_name": "Franca", "capital": "Paris"}
    response = requests.post(api_url, json=country)
    print(response.json())

get()     # lista antes
insert()  # adiciona
get()     # lista depois

Por que usar requests? Em sistemas distribuídos reais, clientes não são humanos no terminal — são outros serviços. requests simula esse papel programaticamente.

Contrato da API

O "contrato" define o que o servidor garante ao cliente: rotas, métodos, formato dos dados e códigos de status. Esta é a documentação que um consumidor precisa para integrar sem ver o código-fonte.

Método Rota Body (JSON) Resposta de sucesso Erros Etapas
GET /countries — 200 — array de objetos — 1, 2, 3
POST /countries {"country_name": "...", "capital": "..."} 201 — objeto criado com id 415 se não-JSON 1, 2, 3
GET /countries/{id} — 200 — objeto 404 não encontrado 3
PUT /countries/{id} {"country_name": "...", "capital": "..."} 200 — objeto atualizado 404, 415 3
DELETE /countries/{id} — 200 — {"message": "País deletado"} 404 3

Formato do objeto Country

Etapas 1 e 2

{
  "id":           1,
  "name":         "Thailand",   // Etapa 1
  "country_name": "Thailand",   // Etapa 2
  "capital":      "Bangkok",
  "area":         513120        // Etapa 1 apenas
}

Etapa 3

{
  "id":           1,
  "country_name": "Thailand",
  "capital":      "Bangkok"
}
Atenção às inconsistências entre etapas: a Etapa 1 usa o campo name, enquanto as Etapas 2 e 3 usam country_name. Em uma API de produção, o contrato seria padronizado desde o início.

Cabeçalhos obrigatórios para POST/PUT

Content-Type: application/json

Sem este cabeçalho, o servidor retorna 415 Unsupported Media Type.

Exercícios

Desafios em ordem crescente de dificuldade. Cada um constrói sobre o anterior.

Fácil — Etapa 1

Validar payload no POST

Em 01-no-db/app.py, adicione validação: se name ou capital não estiverem no JSON recebido, retorne 400 Bad Request com {"error": "Campos obrigatórios: name e capital"}. Teste com curl enviando um body incompleto.

Médio — Etapa 2

Adicionar rota GET /countries/<id>

Em 02-sqlite/app.py, implemente GET /countries/<id> usando db.query_db('SELECT * FROM countries WHERE id = ?', [id], one=True). Retorne 404 se o país não existir. Dica: o argumento one=True em query_db já está implementado em db.py.

Desafio — Etapa 3

Adicionar campo population e endpoint de estatísticas

Em 03-sqlalchemy/app.py: (1) adicione o campo population (Integer) ao modelo; (2) crie o endpoint GET /countries/stats retornando {"total": N, "avg_population": X} usando db.session.query(db.func.count(), db.func.avg(Country.population)).one(). Recrie o banco antes de testar.
